# 量化系统改造规划任务细则（主程序骨架保持版）

> 日期：2026-04-15
> 口径修正：本规划以用户最新澄清为准。
> 核心前提：
> 1. 主程序框架不推倒重来，现有程序继续作为数据底座、工具底座、执行器、监督器、电子围栏。
> 2. 不把 `skills` 当成新的主架构，不围绕 `skills` 重构主程序。
> 3. `skills` 主要装进 Hermes / Agent 运行侧，作为 Agent 可消费工具。
> 4. 真正要改的是：谁来调用这些能力，以及 runtime 如何从“固定筛选器”升级为“可由 Agent 组织参数并调用的工具”。

---

## 一、主基调

本项目的主基调不是“继续增强程序自己做决策”，而是：

- 程序继续做底座。
- Agent 负责决策、协作、提案、调参、调用工具、组织 runtime。
- Hermes 负责承接 Agent 的人格、协作、调度与工具消费。
- runtime 继续保留，但必须增强为 Agent 可主动组织参数、按市场需要调用的证据与筛选工具。

换句话说：

- 程序不是指挥官，而是厨房、仓库、传送带和安检门。
- Agent 才是主厨团队。
- `skills` 不是新的厨房，而是给主厨配的刀具、烤箱、菜单卡和索引手册。

---

## 二、最终目标重述

### 目标 1：主程序定位不变

主程序继续承担以下职责：

- 行情、财务、事件、公告、板块、资金等数据底座
- runtime 扫描、因子计算、证据生成、结构化结果输出
- 下单执行、预检、回执、对账
- 风控边界、黑名单、仓位限制、交易时段限制
- 督办、飞书消息、考核、盘后归因、回测与学习记录

### 目标 2：Agent 成为主动调用者

Agent 应具备以下能力：

- 自主发现市场机会
- 自主决定何时调参数
- 自主决定何时调用 runtime
- 自主组合战法、因子、权重
- 自主提案并接受其他 Agent 质询

### 目标 3：runtime 从“固定选股器”升级为“Agent 工具”

runtime 的角色转为：

- 接收 Agent 组织的参数
- 按 Agent 的问题输出结果
- 返回的是证据、候选、过滤结果、结构化解释
- 而不是每天固定吐一批机械候选股

### 目标 4：Hermes 作为 Agent 运行侧

Hermes 主要承接：

- Agent 的人格与工作方式
- Agent 间协作与委派
- Agent 消费主程序工具
- 外挂 `skills` 的消费
- 盘前、盘中、盘后、夜间的值班编排

---

## 三、当前真正的问题

当前问题不是“程序没有功能”，而是“程序功能虽多，但 Agent 不是主导使用者”。

### 1. 主程序能力已有不少

现有系统已经具备：

- 数据读取
- 运行时扫描
- 执行预检
- 风控拦截
- 讨论主线
- 飞书交互
- 学习与回测骨架

### 2. 真问题是调用权错位

当前仍偏向：

- 程序先产候选
- Agent 再围绕结果解释

而目标应该是：

- Agent 先形成交易假设
- Agent 调程序工具取证
- Agent 调 runtime 验证参数组合
- Agent 再形成正式提案

### 3. runtime 过于僵化

当前 runtime 更像：

- 固定模板粗筛器
- 每次跑出来的票相似度过高
- 和板块热度、新闻催化、盘中势能的关系偏弱

### 4. 监督对象盯错了

当前更像在监督：

- runtime 有没有跑
- 流程有没有走完

目标应改为监督：

- Agent 有没有持续工作
- Agent 有没有新提案
- Agent 有没有根据市场变化调用工具
- Agent 有没有对持仓和热点做响应

---

## 四、改造边界

本轮改造明确 **不做** 以下事情：

- 不重写主程序成全新平台
- 不围绕 `skills` 重构主程序内部架构
- 不把 `skills` 变成新的程序主模块
- 不把 Hermes 做成程序替代品
- 不把 runtime 删除或边缘化

本轮改造明确 **要做** 以下事情：

- 保持主程序骨架稳定
- 给 Agent 更强的参数组织与调用能力
- 给 runtime 更强的参数化、解释性和市场驱动能力
- 给 Hermes 侧接入更多可消费工具
- 改写工作流程，让 Agent 真正成为主动调用者

---

## 五、总体改造主线

## 模块 A：Agent 调用权上移 [P0]

### A1. 明确程序与 Agent 的边界

- 程序负责提供工具
- Agent 负责决定何时、为何、以什么参数调用工具

### A2. 不再由程序默认决定“今天该跑什么”

改成：

- 程序提供固定监测和基础刷新
- 但针对策略性扫描，优先由 Agent 决定是否发起

### A3. 主控负责组织，不垄断结论

主控 Agent 负责：

- 协调角色
- 汇总提案
- 调度工具
- 给出最终对外结论

但不再只是固定 FAQ 中转器。

---

## 模块 B：runtime 工具强化 [P0]

### B1. 新增“Agent 可组织参数调用 runtime”的正式接口

这是本轮最关键任务之一。

必须新增或重构 runtime 调用方式，使 Agent 可以传入：

- 策略类型
- 因子组合
- 因子权重
- 过滤条件
- 板块偏好
- 排除条件
- 风险约束
- 市场上下文
- 候选来源范围

runtime 返回不应只是一组股票，而应包含：

- 候选结果
- 每只标的的证据摘要
- 过滤掉了哪些票，为什么
- 本次参数组合的解释
- 结构化评分拆解

### B1.1 构建“原子化策略仓库”（Strategy Atomic Library）

runtime 不应再只是一个粗糙的统一筛选器，而应升级成一套可编排的策略原子库。

这套原子库必须以“仓库”而不是“几段硬编码逻辑”的形式存在。

也就是说，它必须具备以下特征：

- 可注册：新增因子、新增战法时，不需要改一大片主流程
- 可发现：Agent 能知道当前有哪些可用因子、战法、版本、参数
- 可版本化：同一战法可以存在 `v1/v2/agent-learned-v3`
- 可灰度：新战法、新因子可以先进入模拟评估，再决定是否提升为正式可用
- 可下线：失效策略可以冻结、降权、归档，而不是继续混在正式仓库里
- 可追溯：每个因子、战法、组合模板都能追到来源、作者、版本、表现

这套原子库至少分为五层：

#### 第一层：因子层（Alpha Factory）

把各类因子做成标准化、可组合、可加权的原子能力，而不是散落在固定代码分支里。

应优先模块化的因子包括：

- 动能因子：涨速、趋势斜率、强弱切换、突破延续、回撤修复
- 资金因子：主力净流入、大单占比、封单强度、量比放大、换手质量
- 情绪因子：连板高度、炸板率、涨停溢价、板块扩散、退潮修复
- 技术因子：RSI、MACD、均线结构、波动压缩、放量突破、异动波幅
- 事件因子：政策催化、新闻热度、公告强度、行业事件映射
- 风险因子：流动性、监管风险、财务雷区、解禁压力、异常波动

这些因子应统一具备：

- 标准输入
- 标准输出
- 可解释字段
- 可配置窗口
- 可调权重
- 可回测记录

Agent 不需要改代码，只需要通过 JSON 或结构化参数组合这些因子。

#### 第二层：战法层（Playbook Plugins）

将成熟战法封装成可复用插件，而不是把“某种战法偏好”写死在默认流程里。

不只是一板定龙、弱转强、尾盘潜伏，还应逐步补成更完整的战法族：

- 开盘抢强势
- 分时转强
- 回封确认
- 龙头跟随
- 板块共振
- 低吸修复
- 趋势加速
- 持仓日内 T
- 持仓替换
- 隔夜博弈

每个战法插件至少应暴露：

- 适用市场阶段
- 触发条件
- 必备因子
- 可调参数
- 风险标签
- 证据模板

Agent 可以指定：

- 使用哪个战法
- 战法参数如何调整
- 多个战法如何并行比较
- 哪个战法作为主策略，哪个作为辅助策略

更重要的是，战法插件层必须允许持续扩容：

- 开发侧可以人工新增战法插件
- 研究侧可以把验证过的战法模板沉淀进仓库
- Agent 可以把自己复盘、回测、实盘观察后总结出的新战法，先以候选插件形式写入仓库

但 Agent 学习出来的战法不能直接变成正式默认策略，必须经过：

- 回测或模拟评估
- 讨论质询
- 风控复核
- 审计确认

通过后，才从 `candidate / sandbox / experimental` 提升到 `active`。

#### 第三层：约束层（Constraint Pack）

约束不是战法的一部分，而是 runtime 的硬边界与软偏好层。

约束层至少包含：

- 硬过滤：黑名单、`ST/*ST`、不可交易、涨跌停不可达、风控禁入
- 用户偏好：排除板块、排除主题、仓位上限、单票金额、阶段偏好
- 市场约束：震荡市、退潮期、情绪冰点、监管敏感窗口
- 持仓约束：是否允许替换仓、是否允许做 T、是否允许隔夜持有

这样 Agent 可以大胆组织进攻逻辑，但不会越过程序的电子围栏。

#### 第四层：编排层（Strategy Composer）

runtime 要提供一个策略编排器，让 Agent 可以把“因子 + 战法 + 约束 + 市场上下文”拼成一次正式扫描任务。

它不是简单传一堆参数，而是一次结构化策略表达：

- 当前市场判断
- 目标机会类型
- 候选范围
- 采用的因子集
- 各因子权重
- 采用的战法
- 排除条件
- 风控边界
- 输出排序依据

编排层的意义是让 Agent 真正具备“自己组织 runtime”的能力，而不是只能消费程序默认给出的名单。

#### 第五层：评估层（Evaluator / Backtest Bridge）

任何一次参数组合、战法调用和权重调整，都应天然可进入评估链。

评估层至少记录：

- 本次调用参数
- 触发背景
- 返回候选
- 最终提案
- 是否被采纳
- 后续市场表现
- 失效原因

这样 runtime 不只是一个工具，还会逐步沉淀成 Agent 的进化记录底座。

### B1.2 策略仓库的正式形态

为了避免未来继续把插件能力写散，策略仓库应明确分区：

#### 正式区（active）

用于当前可正式参与 runtime 编排的因子与战法。

#### 实验区（experimental）

用于人工新加但尚未完成验证的因子与战法。

#### 学习区（learned）

用于 Agent 在研究、复盘、夜间沙盘、回测后产出的新策略、新因子组合、新权重模板。

#### 归档区（archived）

用于历史有效但当前失效、降级或不再适配当前市场的插件资产。

策略仓库至少要支持以下元数据：

- `id`
- `name`
- `type`：`factor | playbook | template | learned_combo`
- `status`：`active | experimental | learned | archived`
- `version`
- `author`
- `source`
- `created_at`
- `updated_at`
- `params_schema`
- `evidence_schema`
- `evaluation_summary`
- `risk_notes`

这样以后 runtime 才能像“仓库”一样不断加菜，而不是每次都要回头改主程序骨架。

### B2. runtime 改为“被调用工具”，不是“固定剧本主角”

保留默认流水线，但降级为：

- 基础候选样本生成
- 自动底座刷新

增强为：

- Agent 主动发起参数化扫描
- Agent 主动发起主题扫描
- Agent 主动发起板块联动扫描
- Agent 主动发起持仓替换扫描

### B3. 强化市场驱动输入

runtime 需要补强的输入维度：

- 板块热度
- 龙头映射
- 新闻/政策催化
- 盘中异动
- 势能延续
- 情绪阶段
- 持仓替代价值

### B4. 固化硬过滤

runtime 无论谁调用，都必须先过硬过滤：

- `ST/*ST`
- 黑名单
- 用户排除方向
- 不可交易状态
- 风控硬禁入条件

## 模块 C：Hermes 侧接工具，不改程序骨架 [P0]

### C1. 将外部 `skills` 视为 Agent 工具包

用户提到的那批能力，不应成为主程序大改造中心，而应在 Hermes / Agent 侧作为可消费能力接入。

这些能力可分为三类：

#### 数据与提炼类

- 政策监控
- 研报/财报提炼
- A 股数据源

#### 分析与研究类

- 个股分析
- 量化知识库
- 回测引擎

#### 监督与风险类

- 自选监控
- 风险预警
- 安全审计
- 每日复盘

### C2. Hermes 负责暴露给 Agent

Hermes 侧要做的不是替代程序，而是：

- 给 Agent 注册这些工具
- 让 Agent 知道“能调用什么”
- 把工具说明写进 Agent 提示词与工作协议

### C3. 程序侧只负责提供接口和可调用能力

程序侧要做的是：

- 继续提供数据、runtime、回测、风控、飞书、执行等接口
- 必要时补新接口
- 不强行围绕 `skills` 改主程序结构

---

## 模块 D：参数调节能力重构 [P0]

### D1. Agent 必须能自己组织参数

系统应允许 Agent 通过自然语言或结构化 JSON：

- 调整 runtime 的筛选条件
- 调整因子权重
- 调整过滤逻辑
- 调整策略偏好
- 调整候选来源范围

### D2. 参数变更要有记录与解释

每次参数组织必须留下：

- 谁发起
- 为什么调
- 调了哪些参数
- 对应市场背景是什么
- 后续效果如何

### D3. 参数能力不能只停留在仓位/黑名单

除了现有的：

- 仓位
- 单票金额
- 排除方向

还要补：

- 动量权重
- 事件权重
- 主题权重
- 趋势/反转偏好
- 盘中/隔夜倾向
- 候选规模
- 触发阈值

---

## 模块 E：逻辑辩论机制 [P1]

### E1. Agent A 提案，Agent B 质询

不是只做“支持/反对”，而是：

- 提案方必须附证据
- 反方必须提出反证
- 主控负责收敛

### E2. 风控口一票否决

无论 Agent 多自信，风控硬闸必须保留：

- 黑名单
- 仓位限制
- 回撤限制
- 集中度限制
- 交易时段与流动性限制

### E3. 审计口负责识别“拍脑袋提案”

审计主要判断：

- 证据是否不足
- 是否只在旧池内机械打分
- 是否没有真实市场依据

---

## 模块 F：动态触发机制 [P1]

### F1. 事件驱动触发 Agent，不再机械催 runtime

当以下事件发生时，程序应提醒 Agent：

- 板块 5 分钟快速拉升
- 新闻催化出现
- 持仓突发异动
- 指数情绪切换
- 市场成交额跨阈值

提醒内容是：

- 当前发生了什么
- 哪类 Agent 该介入
- 是否建议组织参数并调用 runtime

### F2. 怠工预警盯 Agent，不盯程序

若市场剧烈变化，但 Agent：

- 没有提案
- 没有调参
- 没有调 runtime
- 没有复核持仓

则判定为“工作迟滞”，而不是去催程序多跑几次。

---

## 模块 G：学习与赛马机制 [P1]

### G1. 回测是评估器，不是装饰物

Agent 调整后的策略或参数，应能进入：

- 微回测
- 虚拟池
- 表现归因

### G2. 建立 Agent 的信誉与权重

每个 Agent 要有：

- 提案命中率
- 调参收益贡献
- 风险违规记录
- 复盘质量

### G3. 形成知识反馈闭环

每次失败都要回答：

- 当时为什么这么调
- 失败在哪
- 以后怎么避免

---

## 六、把用户提到的能力装进 Hermes 的方式

这一部分不是主线，只是接入策略。

### Hermes 侧工具包建议

#### 1. `policy_monitor`

- 监管政策
- 行业新闻
- 市场影响摘要

#### 2. `stock_analyst`

- 个股综合分析
- 技术面、资金面、事件面、板块面

#### 3. `daily_trade_review`

- 盘后复盘
- 持仓得失
- Agent 提案复盘

#### 4. `quant_kb`

- 指标、策略、战法、模型查询

#### 5. `stock_watcher`

- 自选、持仓、热点、异动监控

#### 6. `a_share_data`

- 市场数据统一入口

#### 7. `report_extractor`

- 研报/财报/公告提炼

#### 8. `risk_alert_system`

- 回撤、利空、异常提醒

#### 9. `backtest_engine`

- 策略验证与参数赛马

#### 10. `skill_vetter`

- 权限、安全、数据外发审计

### 接入原则

- 工具装进 Hermes
- Agent 知道如何调用
- 主程序不围绕这些工具重构架构
- 能复用主程序接口的，优先复用主程序

## 七、实施顺序

### 第一阶段：必须先做

1. Agent 可组织参数调用 runtime 的正式接口
2. runtime 结构化结果增强
3. Hermes 侧工具暴露与提示词更新
4. Agent 工作流从“被动接收”改为“主动调用”

### 第二阶段：协作与监督增强

1. 提案与质询机制
2. 事件驱动 Agent 提醒
3. 怠工预警

### 第三阶段：学习与赛马

1. 参数赛马
2. 回测评估

## 八、验收标准

以下条件同时满足，才算贴合目标：

### 1. 程序骨架保持稳定

- 主程序仍是数据、执行、监督、围栏底座
- 没有因为接 skills 而把程序结构改乱

### 2. Agent 真正能自己组织参数并调用 runtime

- 可以通过自然语言或 JSON 组织参数
- 可以主动触发 runtime
- runtime 返回的是带解释的结构化结果

### 3. Hermes 侧工具可消费

- Agent 明确知道有哪些工具
- 工具可被调用
- 工具调用结果能进入提案与讨论

### 4. 监督对象已经转移

- 不是催程序多跑
- 而是催 Agent 对市场变化做响应

### 5. 结果更贴近市场

- 选票不再明显脱离市场热点、板块热度、消息催化和势能
- 选股结果不再只是固定股票池打分
