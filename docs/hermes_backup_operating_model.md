# Hermes 备份控制面协作模式

## 结论

Hermes 可以作为 OpenClaw 的双备份控制面，但不能原样复刻“6 个常驻命名代理同时开会”的模型。更适合本项目的模式是：

- 一个主协调代理
- 两阶段、最多 3 并发的子代理波次
- 用 `cron`、服务支撑和 agent 自主协作承载盘中盯盘、持仓复核、盘后学习、夜间沙盘
- 用 Feishu 作为通知与人工介入入口
- 用程序服务承担真实取数、清洗、执行、留痕、监督和风控边界

## 采用该模式的依据

基于 Hermes 官方文档，本项目必须接受以下事实：

1. `delegate_task` 最多并发 3 个子代理，超过 3 个不会稳定扩展。
2. 子代理是全新上下文，只知道父代理传入的 `goal/context`。
3. 子代理不能递归再委派，也不能直接做跨平台消息副作用。
4. `cron` 每次执行都是全新会话，适合做值班任务，不适合依赖上次会话记忆。
5. 提示词层级应分为：
   - `SOUL.md`：稳定人格
   - `.hermes.md` / `HERMES.md`：项目规则
   - 角色提示词模板：委派时按需注入

这意味着 Hermes 的正确打开方式不是“养 6 个长期在线人格”，而是“主协调按波次拉起角色化工作单元”。

## 对应到 A 股交易台的最优编排

### 主代理

主代理负责：

- 读取仓位目标、系统 cadence、服务侧 review 入口、当前候选与持仓态
- 基于服务事实自主判断当前该优先处理“补仓审查、持仓复核、替换审查、盘后学习”中的哪一类
- 决定委派给哪些角色
- 汇总结构化返回并形成最终结论

主代理不应把自己降级为纯打分器，也不应跳过子代理直接编造专业结论。
主代理可以自主决定何时发起研究、讨论和提案建议，但所有正式流程都必须通过服务接口留痕并接受监督。

### 波次 A：机会发现

并发 3 个角色：

1. `runtime_scout`
   - 看 runtime、monitor、服务生成的候选/监控工单与新机会信号
   - 负责“盘面事实”
2. `event_researcher`
   - 看新闻、公告、政策、题材、逻辑破坏
   - 负责“事件事实与催化边际”
3. `strategy_analyst`
   - 比较候选、持仓与外部新票
   - 负责“相对强弱、替换价值、日内 T 价值”

### 波次 B：准入收敛

1. `risk_gate`
   - 给 `allow / limit / reject`
   - 若未满仓且不放行，必须说清保留空仓的原因
2. `audit_recorder`
   - 检查证据链、提票规范、是否存在组织失误
3. `execution_operator`
   - 仅在执行意图已成型、且需要预演或真实提交时介入

## 为什么这比照搬 OpenClaw 更优

### 更符合 Hermes 原生能力

- 波次式并发正好贴合“最多 3 并发”上限。
- 主代理保留全局态势，子代理只拿最小必要上下文，减少上下文污染。
- `cron` 天然适合“盘中持续巡检”和“盘后每天都做研究与学习”。

### 更接近真实交易台

- 不是“等会议开始才工作”，而是 agent 持续工作，服务持续提供数据、工具、监督和电子围栏。
- 新机会提案可以由 agent 主动发起，但必须通过服务固化、留痕和接受风控约束。
- 满仓后也允许替换仓位，不把“仓已满”当作停止思考的借口。

## 建议的日内运行流

### 盘前

- 预热 dossier 与 workspace context
- 检查目标仓位、隔夜持仓风险、竞价窗口任务

### 盘中

循环执行：

1. 主代理读取系统状态、服务待办与最新数据，自主判断当前该推进什么
2. 波次 A 并发分析服务给出的候选、持仓复核单和新机会线索，也可对新线索主动发起提案建议
3. 若出现重点新票、持仓异动或替换候选，再跑波次 B
4. 需要执行预演或提交时再叫执行代理
5. 把结果回流给当前会话或 Feishu

### 盘后

- 复盘当日 missed opportunities、late discoveries、risk overblocking、unsupported promotions
- 把能固化的经验写入学习材料、治理提案或 prompt patch 候选

### 夜间

- 运行 nightly sandbox
- 回看 watchlist 与替换决策是否有更优参数
- 产出次日需重点盯盘的题材、个股和仓位建议

## 提示词与配置落点

本仓库已采用如下落点：

- 项目规则：`/.hermes.md`
- 协作剧本：`/docs/hermes_backup_operating_model.md`
- 角色提示词：`/hermes/prompts/*.md`
- 后续可在 profile 级写 `SOUL.md`，提供更稳定的“交易台人格”

## Feishu 与自然语言调参

Hermes 官方 Feishu 网关支持：

- WebSocket 模式
- home channel 投递 cron 结果
- 群聊按用户隔离 session

本项目建议：

1. 默认使用 WebSocket 模式。
2. 开启 `group_sessions_per_user=true`。
3. 把调参结果、盘后学习摘要、夜间沙盘摘要优先投到 Feishu home channel。
4. 用户口语化调参优先走 `/system/adjustments/natural-language`，不要在提示词里手工模拟 proposal。

## 服务边界补充

- Windows/QMT 是原始数据源，Hermes 不直接承担跨平台采集职责。
- 本项目程序本体负责把 Windows 侧数据清洗成 runtime、monitor、dossier、review gate、执行接口与考核接口，并承担留痕、监督和风控边界。
- Hermes 的职责是消费这些接口，产出分析、质疑、排序、风险和摘要，并自主组织团队协作。
- 对候选池外发现的新线索，Hermes 可以主动形成结构化 `opportunity_ticket` 提案；程序服务负责把它固化、监督、打分、执行或拦截。
