# Agent 自主编排与 101 因子库实施细则

## 1. 目标重述

本项目下一阶段不再以“程序固定流程”作为中心，而是正式切换为：

- Agent 是交易团队的大脑，负责理解市场、提出假设、组织工具、组合战法、发起质询、生成执行建议。
- 程序是手脚、工具菜单、电子围栏、审计与监督系统，负责提供数据、算子、执行能力与可追溯治理。
- Runtime 不再是固定筛选器，而是一个可被 Agent 主动消费的原子策略仓库。

本阶段核心目标不是“再多补几个接口”，而是把以下三件事做实：

1. Agent 真正具备“理解市场后再选工具”的能力。
2. Runtime 真正具备“可自由组合的因子库 + 战法库”能力。
3. 监督系统真正盯的是 Agent 的市场响应与产出质量，而不是机械工具调用次数。

## 2. 当前真实现状

当前系统已经具备以下底座：

- 已有 `compose-from-brief` 契约，可承载 `playbooks / factors / weights / constraints / market_hypothesis`。
- 已有 `runtime-work-packets` 与 supervision task plan，能向 agent 分发任务、角色边界、读写接口与 compose 起手提示。
- 已有原子注册表：
  - 因子注册表 [factor_registry.py](/srv/projects/ashare-system-v2/src/ashare_system/strategy/factor_registry.py)
  - 战法注册表 [playbook_registry.py](/srv/projects/ashare-system-v2/src/ashare_system/strategy/playbook_registry.py)
- 已有初始种子：
  - 因子约 14 个
  - 战法约 10 个
- 已有监督侧自然语言任务提示，已经开始强调“先形成 compose 草案，再决定是否调用 runtime”。

但当前仍有三个结构性不足：

1. Agent 当前主要还是“从预置 profile 中选起手”，而不是自己拼 compose。
2. 候选源仍偏依赖既有 candidate pool 和基础排序，主动发现新票的能力不足。
3. 因子库与战法库规模太小，尚未达到多因子均值回归和多风格赛马的有效规模。

## 3. 设计原则

### 3.1 Agent 优先

- 程序不直接替 Agent 做交易判断。
- 程序只提供：
  - 市场数据
  - 新闻与事件摘要
  - 因子与战法菜单
  - compose 校验与执行
  - 风控与审计闸门

### 3.2 原子化

- 不再提供“一个大筛选器”。
- 所有策略能力拆成：
  - 原子因子
  - 原子战法
  - 可组合约束
  - 可追踪权重
  - 可复用模板

### 3.3 事件驱动

- Runtime 不是定时器驱动优先，而是事件驱动优先。
- 触发来源包括：
  - 市场异动
  - 板块升温/退潮
  - 持仓变化
  - 尾盘窗口
  - 新闻/政策催化
  - Agent 主动提案

### 3.4 赛马与淘汰

- 因子、战法、agent 组合都必须纳入账本与赛马。
- 不再允许长期存在“没人知道是否有效”的默认策略。

## 4. 总体架构

下一阶段主线改造成 5 层。

### L1 市场理解层

由 Agent 负责。

输出：

- 当前市场阶段
- 主线/支线板块
- 风险偏好
- 交易重点
- 持仓应对动作

输入工具：

- 行情快照
- 板块热度
- 新闻/政策事件
- 持仓状态
- 执行与风控状态

### L2 候选发现层

由程序提供多入口，Agent 自主选择使用。

候选入口至少包含：

- 全市场粗扫描入口
- 板块热点入口
- 新闻催化入口
- 持仓异动入口
- 日内做 T 入口
- 尾盘潜伏入口
- Agent 自提机会票入口

### L3 Compose 编排层

由 Agent 输出 compose brief。

compose brief 至少包括：

- 市场假设
- 目标任务
- universe 来源
- symbol pool / sector focus
- playbooks
- factors
- weights
- constraints
- risk rules
- 执行理由
- 放弃项

程序负责：

- 校验 schema
- 落盘 trace
- 触发 runtime compose
- 产出评估记录

### L4 风控与执行层

由程序负责，但只做边界控制与执行，不替 Agent 决策。

内容包括：

- 仓位限制
- 单票金额限制
- 黑名单/偏好排除
- 流动性与涨跌停约束
- 实盘/预演闸门
- 执行回执

### L5 学习与治理层

对以下对象形成账本与评估：

- agent
- playbook
- factor
- compose 组合
- 执行结果
- 次日复盘结论

## 5. Agent 自主编排改造要求

### 5.1 从“选 profile”升级为“自组装 compose”

当前 `profile` 保留，但只作为参考模板，不再作为唯一入口。

改造要求：

- Agent 可以完全绕过默认 profile，自行组织：
  - playbooks
  - factors
  - weights
  - constraints
- 程序只负责校验字段完整性与合法性。
- Runtime 必须接受“任意合法 compose brief”，而不是只接受几个固定模板。

### 5.2 Agent 提示词重构要求

各 agent 的提示词必须明确写出：

- 你先理解市场，再决定是否调用工具。
- 不允许把固定股票池排序结果当成最终结论。
- 当市场变化明显时，应主动调整权重、战法和 universe。
- 当持仓存在时，应优先考虑：
  - 去弱留强
  - 换仓
  - 做 T
  - 尾盘动作
- 若默认数据不足以解释机会，可以调用其他工具或发起新提案。

### 5.3 Agent 输出必须结构化

每次关键编排都要求输出：

- 市场假设
- 选择该工具组合的理由
- 放弃其他工具组合的理由
- 是否需要 runtime 验证
- 验证后下一步动作

## 6. 候选发现层改造要求

### 6.1 候选源拆成多入口

必须新增或明确区分以下 source：

- `market_universe_scan`
- `sector_heat_scan`
- `news_catalyst_scan`
- `position_monitor_scan`
- `intraday_t_scan`
- `tail_ambush_scan`
- `agent_proposed`

### 6.2 Agent 自提机会票机制

Agent 有权直接提交新票，不要求必须先在默认股票池中出现。

提交内容至少包括：

- symbol
- 发现来源
- 市场逻辑
- 核心证据
- 风险点
- 为什么值得进入讨论

### 6.3 候选池分层

候选池至少拆成三层：

- 粗样本池
- 深度验证池
- 执行候选池

避免当前“所有东西都混在一个 candidate pool 里”的情况。

## 7. 101 因子库实施要求

### 7.1 总体原则

101 因子不是神秘数字，而是第一阶段规模目标。

目标不是追求 101 个名字，而是建立：

- 足够异质的因子族群
- 因子元数据
- 因子评估账本
- 因子淘汰与复活机制

### 7.2 第一阶段目标

第一阶段目标建议：

- 因子总数扩至 `48-64`
- 战法总数扩至 `18-24`
- 每个因子必须具备：
  - `id`
  - `group`
  - `version`
  - `description`
  - `params_schema`
  - `evidence_schema`
  - `executor`
  - `source`
  - `author`
  - `status`

### 7.3 因子分组

第一阶段建议至少包含以下 10 组：

1. 趋势动量组
2. 反转修复组
3. 成交量能组
4. 资金行为组
5. 板块热度组
6. 事件催化组
7. 微观结构组
8. 风险惩罚组
9. 估值排雷组
10. 持仓管理组

### 7.4 第一阶段建议补齐的因子方向

在现有 14 个基础上优先补：

- 多周期收益率：1d / 3d / 5d / 10d / 20d
- 多周期斜率：5d / 10d / 20d
- 均线发散度
- 近端突破强度
- 缩量回踩质量
- 放量突破确认度
- 换手率分位数
- 量价背离
- 主动买卖强度
- 大单净额斜率
- 封单稳定度
- 连板梯队热度
- 同题材扩散度
- 新闻密度
- 公告催化等级
- 高开承接质量
- 低开反包质量
- 尾盘抢筹强度
- 做 T 适配度
- 持仓替换性价比
- 个股波动惩罚
- 跳空风险
- 流动性断层风险
- 监管/ST/解禁排雷

### 7.5 因子治理要求

每个因子必须进入账本，记录：

- 使用次数
- 命中标的
- 命中市场阶段
- 收益贡献
- 风险贡献
- 与其他因子相关性
- 最近 5/20/60 日有效性

## 8. 战法仓库实施要求

### 8.1 战法定位

战法不是大而全模板，而是 Agent 可点选的“原子战术”。

### 8.2 第一阶段建议扩充方向

除现有 `sector_resonance / weak_to_strong / dragon_returns / oversold_rebound` 等外，优先补：

- 一板定龙
- 二波新高
- 弱转强确认
- 高开承接
- 低开反包
- 尾盘潜伏
- 分时卡位
- 板块分歧转一致
- 跟风剔除
- 龙头换手接力
- 持仓做 T
- 持仓替换
- 回调低吸
- 首阴反包
- 消息催化抢反应

### 8.3 战法账本

每个战法记录：

- 适用市场阶段
- 使用次数
- 入选率
- 预演通过率
- 实盘触发率
- 收益/回撤
- 最近失效模式

## 9. Runtime 改造要求

### 9.1 Runtime 角色重定义

Runtime 不再定义为“选股器”，而定义为：

- 原子因子求值器
- 原子战法求值器
- Compose 编排执行器
- 多候选评分器
- 约束应用器

### 9.2 Runtime 输入

必须支持：

- 任意 compose brief
- 多 source 候选
- 多 market context
- 持仓上下文
- 用户偏好与自然语言调参结果
- 风控规则与执行边界

### 9.3 Runtime 输出

除候选结果外，必须输出：

- 使用的 playbooks
- 使用的 factors
- 权重表
- 被过滤的原因
- score breakdown
- evidence lines
- trace_id

## 10. 监督系统改造要求

监督系统不再重点盯：

- 是否按时调用 candidate_poll
- 是否固定调用某个 runtime 接口

而重点盯：

- Agent 是否在市场关键节点有新产出
- 是否根据市场变化调整了假设
- 是否形成新的 compose brief
- 是否提出了新机会票
- 是否对持仓采取了应对动作
- 是否完成盘后复盘与学习沉淀

## 11. 分阶段实施顺序

### P0：把主线从“模板驱动”改成“Agent 自组装驱动”

- 打通 agent 自组装 compose brief
- 保留 profile 但降级为参考模板
- 输出结构化编排记录
- 把新票提案入口接入 discussion

### P1：扩容原子因子库与战法库

- 因子扩到 48-64
- 战法扩到 18-24
- 建立因子/战法元数据与账本

### P2：引入赛马与淘汰

- 因子赛马
- 战法赛马
- compose 组合赛马
- regime 分桶评估

### P3：让 Agent 能基于历史学习主动生成新组合

- 复用盘后学习结果
- 生成 learned combo
- 次日优先推荐学习有效组合

## 12. 交付验收标准

达到以下条件，才算这一阶段主线完成：

1. Agent 能在无固定 profile 的前提下，自主提交合法 compose brief。
2. Agent 能从非默认股票池中提出新机会票，并进入讨论。
3. Runtime 因子库达到第一阶段规模目标，且全部纳入注册表与账本。
4. Runtime 输出能清楚说明本次使用了哪些因子、战法与权重。
5. 监督系统能识别“市场有变化但 Agent 无产出”的真实怠工。
6. 盘中、尾盘、持仓做 T、换仓、盘后学习都能走同一套 Agent 主导逻辑，而不是碎片化脚本。

## 13. 最终判断

当前项目已经具备“Agent 中心化改造”的骨架，但还没有达到“Agent 真正理解市场并自主组织工具”的标准。

本阶段真正的主线不是继续修零散接口，而是：

- 扩大 Runtime 的工具供给
- 把候选发现改为多入口
- 让 Agent 从“拿模板”升级为“自己拼工具组合”
- 用账本和赛马把因子、战法、Agent 全部纳入可追踪治理

这一步做完，系统才会从“会跑流程”进入“会做交易研究和策略组织”。
