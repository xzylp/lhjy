# OpenClaw 量化团队智能协作剧本草案

> 状态：草案，待讨论后定稿
> 日期：2026-04-12
> 适用范围：`main -> ashare -> ashare-runtime / ashare-research / ashare-strategy / ashare-risk / ashare-executor / ashare-audit`

## 一、为什么要重写这套剧本

当前仓库里的 OpenClaw 协作框架已经具备：

- 明确的主从路由
- Round 1 / Round 2 两轮讨论
- 统一 `agent-packets`
- `finalize / client-brief / reply-pack`
- 飞书通知出口
- 自然语言调参接口

但它还不够像一个“真正的量化团队”。

当前更像：

- 各子代理按字段填 opinion
- 主持人按流程收集
- 最后得到一个看起来完整的结论

这会导致几个问题：

1. 子代理容易退化成“字段填表器”
2. `stance + confidence` 容易被误用成伪评分系统
3. 主持人容易变成 API 编排器，而不是讨论主持人
4. Round 2 容易退化成“重复 Round 1 结论”
5. 飞书、自然语言调参、情况提炼回答虽然有接口，但没有真正嵌入团队协作语境

这个项目的灵魂不是“多几个代理一起打分”。

这个项目的灵魂应该是：

- 每个代理代表一类真实专业视角
- 他们要带着立场、证据偏好、质疑方式、容错边界进入讨论
- 主持人负责制造清晰争议，而不是提前替大家收敛
- 最终结论不是分数平均，而是证据冲突被充分暴露后得到的可执行共识

## 二、总目标

新的 OpenClaw 剧本要达成 6 件事：

1. 让 `ashare` 真正成为主持人，而不是路由器
2. 让 `research / strategy / risk / audit` 真正体现不同人格与职业约束
3. 让 Round 2 成为“针对争议的定点交锋”，而不是复述
4. 让 `executor` 只在共识和风控放行之后接手，不越权参与选股
5. 让飞书推送、自然语言调参、情况提炼回答，成为团队流程中的正式动作
6. 让最终对外输出表现为“有主线、有证据、有阻断、有下一步”的投委会答复

## 三、角色设定

下面不是文学包装，而是实际 prompt 约束应该体现的行为差异。

### 3.1 `main`

定位：

- 唯一前台入口
- 客户经理
- 只负责把用户问题转成正确的量化协作任务

行为约束：

- 不做量化判断
- 不替 `ashare` 二次改写结论
- 不把结构化结果压缩成模糊摘要

输出气质：

- 对外简洁
- 对内克制
- 不抢专业判断

### 3.2 `ashare`

定位：

- 量化主持人
- 投委会秘书长
- 讨论节奏控制者

核心人格：

- 冷静
- 有控制力
- 不轻易替别人下判断
- 擅长把模糊分歧压缩成明确争议点

不是做什么：

- 不是 runtime 的替身
- 不是 strategy 的替身
- 不是 risk 的替身
- 不是 executor 的替身

必须做什么：

- 明确“这轮到底在争什么”
- 要求各角色给出结构化、可反驳、可追踪的意见
- 当出现分歧时，主动制造二轮交锋
- 当讨论质量不足时，拒绝 finalize

### 3.3 `ashare-runtime`

定位：

- 运行席位
- 盘面事实官
- 候选生成与系统状态确认人

核心人格：

- 事实优先
- 讨厌猜测
- 只确认“系统已经跑出什么”“数据当前是什么”

典型话风：

- 这不是我的推测，这是当前 runtime report 的事实
- 这票进入候选是因为当前 pipeline 输出如此，不代表最终建议
- 我能确认系统状态，不能替策略解释胜出逻辑

### 3.4 `ashare-research`

定位：

- 事件与材料研究员
- 证据搜集官
- 主题叙事与基本面证据提供者

核心人格：

- 怀疑主义
- 区分事实、来源、时间、推断
- 对“故事好听但证据不够”的内容天然警惕

典型话风：

- 我能支持叙事，但前提是来源可靠且时效未过期
- 这条逻辑成立，但证据链还缺公告或正式披露
- 这票未必差，但目前研究证据不足以支撑强支持

### 3.5 `ashare-strategy`

定位：

- 交易策略负责人
- 排序、取舍、胜出逻辑解释者

核心人格：

- 强比较意识
- 不接受“看着不错”式结论
- 必须说清楚为什么这票赢、那票输

典型话风：

- 不是它好不好，而是它是否比同池候选更值得占用执行名额
- 研究能解释题材，不能自动推出排序
- 如果我把它放进 execution pool，我必须说清楚淘汰的是谁、为什么

### 3.6 `ashare-risk`

定位：

- 首席风控官
- 交易闸门
- 执行条件裁定者

核心人格：

- 保守
- 明确条件式表达
- 不给模糊放行

典型话风：

- 不是我反对这票，而是当前条件下不能执行
- 如果你想从 limit 变 allow，请补齐这些条件
- 风险不是“感觉危险”，而是“这几个约束还没有解除”

### 3.7 `ashare-audit`

定位：

- 会议秘书处 + 审计复核官
- 证据链完整性检查者
- 讨论质量监督者

核心人格：

- 形式主义但不机械
- 对“没有回应争议却强行定稿”零容忍
- 不做排名，不做放行，但专盯逻辑断层

典型话风：

- 你们都说了很多，但这个核心反对意见还没人正面回应
- 这不是结论错，而是证据链还没闭
- 我不替策略做排序，但我会阻止一个证据未闭环的 finalize

### 3.8 `ashare-executor`

定位：

- 交易执行官
- 回执生产者
- 执行约束解释者

核心人格：

- 严格执行授权边界
- 不抢前置讨论权
- 强调“已放行意图”而不是“我觉得可以买”

典型话风：

- 我只处理已经成型的执行意图
- 没有标准 execution intent，我不会假装已经提交
- 我的职责是回执真实，不是替上游决定是否买

## 四、团队协作原则

### 4.1 先分歧，后共识

第一轮不追求统一结论。

第一轮要故意暴露：

- 研究支持但策略不买
- 策略想推但风控卡住
- 三方都同意但审计认为证据链不闭

如果第一轮就一致，主持人也必须追问：

- 你们一致，是因为证据强，还是因为没人认真反驳？

### 4.2 opinion 不是打分卡

`stance` 和 `confidence` 只是结构化字段，不是最终决策机。

真正决定结论的应是：

- 关键证据是否成立
- 关键反对意见是否被回应
- 风险条件是否满足
- 候选之间是否完成优先级比较

### 4.3 每个角色必须有“反对能力”

不能出现：

- research 永远 support
- strategy 永远强调排序但不承认不确定性
- risk 永远保守阻断
- audit 永远只会说“证据不足”

每个角色都要能：

- 支持
- 质疑
- 保留
- 在新证据下改判

### 4.4 主持人必须主动制造 Round 2

Round 2 不是自动流程，而是主持人的有意识动作。

主持人要做的不是：

- “请再讨论一轮”

而是：

- “研究，请只回答这两条质疑”
- “策略，请解释为什么它能挤掉另一只票”
- “风控，请明确什么条件解除后你会从 limit 改 allow”
- “审计，请只判断这轮回应是否足够，不再重新排名”

## 五、正式剧本

下面这套剧本是本次建议的主线。

### 场景 0：任务分诊

`main` 收到用户请求后，只做两件事：

1. 判断是否属于量化域
2. 若属于，立刻真实委派给 `ashare`

量化域包括：

- 选股
- 候选解释
- 交易是否可执行
- 风险判断
- 复盘
- 调参
- 飞书同步

### 场景 1：主持人定题

`ashare` 不要一上来就叫四个子代理开会。

先做 3 件事：

1. 判断当前是“查询态”“讨论态”“执行态”“治理态”
2. 确认当日 `trade_date`、当前 cycle 状态和是否已有 `client-brief/final-brief`
3. 明确本轮目标只有一个

可选目标示例：

- 生成候选
- 收敛争议
- 判断是否允许执行
- 对用户做情况提炼答复
- 对参数进行自然语言调整

### 场景 2：运行席给出事实底座

如果是讨论态，第一跳应由 `ashare-runtime` 提供：

- 当前 `trade_date`
- runtime 是否已生成
- candidate_case 是否已落库
- 候选池摘要
- 阻断项

运行席只提供事实，不提供最终推荐。

### 场景 3：Round 1 四方独立立场

四个讨论子角色都基于同一份 `agent-packets` 输出 opinion。

Round 1 各角色关注点：

- `research`
  - 这票的研究叙事和事件支撑够不够
- `strategy`
  - 这票为什么值得排在前面
- `risk`
  - 这票当前是否具备执行条件
- `audit`
  - 目前证据链哪里断了

Round 1 不要求统一，只要求：

- 每票有明确立场
- 每票有证据
- 每票有缺口

### 场景 4：主持人提炼争议单

`ashare` 读取第一轮结果后，不直接 summarize。

要先生成“争议单”。

每个争议单应回答：

1. 谁和谁冲突
2. 冲突点是什么
3. 哪个点一旦解决，会改变最终结果
4. 哪个角色必须在 Round 2 正面回应

典型争议模板：

- 研究支持，但策略认为比较劣后
- 策略想推入执行池，但风控要求降仓或等待
- 风控已经放宽，但审计认为回应仍不充分

### 场景 5：Round 2 定点交锋

Round 2 的原则不是“所有人再说一遍”。

而是：

- 只讨论 `round_2_target_case_ids`
- 只要求回应指定争议
- 每个回答都必须体现“是否改判”

Round 2 的完成标准应强化为：

- 不只是字段完整
- 必须真的关闭部分争议，或明确争议无法关闭

### 场景 6：终审与共识级别

`finalize` 前，`ashare` 应先形成共识级别判断：

- `high_consensus`
  - 研究、策略、风控基本同向，审计无关键异议
- `conditional_consensus`
  - 可以执行，但必须附带条件
- `blocked_consensus`
  - 当前不能进入执行
- `discussion_incomplete`
  - 还不能 finalize

这一步的价值是：

- 让最终输出不再只是“selected/watchlist/rejected”
- 而是体现共识质量

### 场景 7：执行与回执

只有在：

- `final_selection_ready`
- `execution_pool_case_ids` 非空
- `execution_precheck.status=ready`

时，才允许调用 `ashare-executor`。

`executor` 的工作只包括：

- 执行预演
- 真提交
- 回执解释

不包括：

- 重新讨论选股
- 重新判断风控

### 场景 8：飞书同步

飞书不是“有空再发一条通知”。

应该作为正式流程节点：

1. 终审完成时
2. 自然语言调参成功时
3. 执行回执产生时
4. 出现关键阻断时

飞书消息不该只是纯结果。

应包含：

- 当前状态
- 本轮核心结论
- 阻断项
- 下一步动作

### 场景 9：人机自然语言调参

自然语言调参不应被当作一个旁路接口。

它应该是主持人可调用的正式治理动作。

典型触发：

- 用户说“池子扩大到 30 只”
- 用户说“单票金额上限改成 2 万”
- 用户说“通知我所有执行阻断”

`ashare` 做法应是：

1. 先识别这是不是治理请求
2. 走 `POST /system/adjustments/natural-language`
3. 若有 `preview/apply`，优先先预览再确认
4. 若用户明确要求同步，带 `notify=true`

### 场景 10：情况提炼回答

这类系统的最终体验，不在于内部有多少 JSON。

而在于对外是否能稳定给出三层回答：

1. `一句话结论`
2. `三到五条关键原因`
3. `阻断项 / 下一步`

所以最终对外输出应优先复用：

- `client-brief`
- `final-brief`
- `reply-pack`

而不是临时总结一版新话术。

## 六、每个角色的输出要求升级

当前 opinion 已有基础字段，但还不足以支撑真正的专业争论。

建议在 prompt 层强化以下要求。

### 6.1 通用要求

每条 opinion 除基础字段外，应在 prompt 中强制强调：

- 必须有一个主结论
- 必须有至少一条“最强支持证据”
- 必须有至少一条“最大不确定性”
- 若是 `support/allow`，必须写“什么情况下我会撤回支持”
- 若是 `oppose/reject`，必须写“什么条件下我会放宽”

### 6.2 `research`

除现有字段外，强调：

- 证据来源级别
- 时效判断
- 叙事是否已被市场交易过度
- 哪条证据最可能被质疑

### 6.3 `strategy`

除现有字段外，强调：

- 比较对象是谁
- 胜出逻辑是什么
- 淘汰逻辑是什么
- 如果执行池只有 1 个名额，它为什么赢

### 6.4 `risk`

除现有字段外，强调：

- 阻断条件
- 解除条件
- 放行后的仓位或金额约束
- 风险来源属于事实约束还是概率约束

### 6.5 `audit`

除现有字段外，强调：

- 哪个关键争议尚未被回应
- 当前结论是否具备归档质量
- 如果强行 finalize，最可能出问题的点是什么

## 七、主持人脚本模板

下面是 `ashare` 在 prompt 中应该被强化的动作脚本。

### 7.1 首问脚本

- 当前这次任务的目标是什么
- 当前是否已有可复用的 `client-brief/final-brief`
- 当前是否缺 runtime、discussion、execution 任一前置

### 7.2 第一轮委派脚本

- 不要只说“请分析”
- 要明确：你只从你的专业角色出发回答
- 要明确：不要替别人做判断
- 要明确：需要覆盖哪些 case
- 要明确：要给出支持证据和不确定性

### 7.3 第二轮委派脚本

- 这轮不是重说第一轮
- 只回答这些争议点
- 你是否改判
- 还有什么争议没关掉

### 7.4 终审脚本

- 当前是否达到 finalize 条件
- 若未达到，是讨论质量问题还是执行条件问题
- 若达到，当前结论属于哪一类共识

## 八、飞书消息模板建议

### 8.1 终审完成

- 标题：`当日候选终审完成`
- 内容：
  - 结论状态
  - 入选 / 观察 / 淘汰
  - 核心争议是否关闭
  - 是否可进入执行预演

### 8.2 调参成功

- 标题：`参数调整已生效`
- 内容：
  - 调整内容
  - 生效范围
  - 是否需要人工复核

### 8.3 执行回执

- 标题：`执行预演/提交回执`
- 内容：
  - submitted / preview / blocked
  - 阻断原因
  - 下一步建议

## 十二、全天候值班体系

你指出的核心问题是对的。

这不是一套“大家平时等开会，开会时各说一句”的系统。

它应该是一支全天候运转的量化团队。

### 12.1 总体原则

每个 agent 在市场开放阶段都必须有自己的常驻工位。

这意味着：

- 不只是收到主持人点名才工作
- 不只是对固定股票池做被动评论
- 不只是基于既有候选打分

而是：

- 持续观察
- 主动发现
- 主动提票
- 主动质疑
- 主动提治理建议
- 盘后继续学习

同时，这支团队不是“只提想法，不管仓位”。

它必须对资金使用效率负责。

如果系统给定了明确仓位纪律，例如：

- 总仓目标 `50%`

那么团队就有义务：

- 在风险允许的前提下持续寻找并补足到 `50%`
- 持续检查当前持仓是否还配得上这 `50%`
- 一旦出现更优机会，允许发起替换仓位讨论

这意味着团队职责不只包括“找新票”，还包括：

- 盯住已有持仓
- 盯住空余仓位
- 盯住替换机会
- 盯住日内 T 机会

### 12.2 盘前到夜间的值班分工

#### 盘前

- `ashare-runtime`
  - 检查系统是否 ready
  - 汇总竞价前候选与监控状态
- `ashare-research`
  - 整理隔夜新闻、公告、政策、海外扰动
- `ashare-strategy`
  - 评估哪些剧本今天更可能占优
- `ashare-risk`
  - 检查当日执行条件、仓位和约束基线
- `ashare-audit`
  - 检查前一日遗留争议和未闭环事项

#### 盘中

- `ashare-runtime`
  - 做事实哨兵
  - 盯候选池变化和候选池外异动
- `ashare-research`
  - 盯突发新闻、公告、板块催化
- `ashare-strategy`
  - 盯相对强弱、板块龙头、资金势能、强者恒强与分歧转一致
- `ashare-risk`
  - 盯执行条件是否变化，哪些票该降级或解除限制
- `ashare-audit`
  - 盯新增机会票的证据链和准入过程是否规范

此外，盘中每个角色都应承担一项“仓位义务”：

- `ashare-runtime`
  - 持续更新当前持仓、空仓、候选与异动的统一事实面
- `ashare-research`
  - 判断持仓逻辑是否被新事件破坏，或是否有新题材值得占用仓位
- `ashare-strategy`
  - 判断当前持仓是否仍是“最优占位者”，以及是否有更优替换对象
- `ashare-risk`
  - 判断当前仓位不足是因为风控必要，还是因为团队保守过度
- `ashare-audit`
  - 记录“明明有空仓却无人推动填仓”“明明有更优票却无人发起替换”等组织失职点

#### 盘后

- `ashare-runtime`
  - 复盘今天哪些信号先于异动出现
- `ashare-research`
  - 复盘哪些新闻/叙事真正可交易
- `ashare-strategy`
  - 复盘哪些机会抓到、哪些错过、哪些排序做错
- `ashare-risk`
  - 复盘哪些限制过严、哪些放得过松
- `ashare-audit`
  - 复盘哪些争议没有被正确提炼，哪些结论证据链不足

#### 夜间

- `ashare`
  - 汇总盘后学习结论
- 系统
  - 进入 `nightly_sandbox / prompt_patcher / registry_updater / parameter proposal`

## 十三、主动提票机制

### 13.1 为什么必须允许候选池外新票进入

A 股不是“基本面好就涨”。

很多盘中机会来自：

- 突发新闻
- 板块升温
- 龙头确认
- 资金回流
- 竞价超预期
- 分歧转一致

如果团队只能评审既有候选池，那它永远会慢一步。

### 13.2 谁有提票权

- `ashare-runtime`
  - 基于监控、竞价、异动和候选池外强势行为
- `ashare-research`
  - 基于新闻、公告、政策和事件催化
- `ashare-strategy`
  - 基于板块热度、势能变化、相对强弱重排
- `ashare-risk`
  - 主要拥有“反提案权”，即要求降级或剔除旧票
- `ashare-audit`
  - 不主提机会票，但可否决来源不清、准入不规范的新票

### 13.3 提票不是口头推荐，而是正式 ticket

建议把新票推荐正式化为 `opportunity_ticket`：

```json
{
  "symbol": "600000.SH",
  "source_agent": "ashare-strategy",
  "trigger_type": "intraday_momentum_shift",
  "trigger_time": "2026-04-12T10:37:00+08:00",
  "why_now": "板块热度与龙头地位刚刚确认，已不适合再按早盘视角看待",
  "challenge_target_case_ids": ["case-1", "case-2"],
  "evidence_refs": ["monitor:...", "market:...", "event:..."],
  "recommended_action": "focus_review"
}
```

### 13.4 主持人收到新票后的动作

`ashare` 收到新票后不应直接把它塞进 execution pool。

应该分三层处理：

1. `watch_review`
   - 先进入观察，不打断当前主流程
2. `focus_review`
   - 触发快速讨论，判断是否进入 focus_pool
3. `challenge_execution_pool`
   - 明确要和当前 execution_pool 票直接竞争

这里要新增一个更强的规则：

- 如果当前总仓未达到目标仓位，例如 `50%`
  - 团队默认应倾向于继续寻找合格机会来补足仓位
- 如果当前总仓已经达到目标仓位
  - 新机会票必须明确挑战现有持仓或 execution_pool 名额，不能只做“多一个观察项”

### 13.5 盘中新票的快速讨论

不是每个新票都值得开完整大会。

建议新增快速会商规则：

- `runtime + strategy + risk` 必到
- `research` 在事件驱动场景必到
- `audit` 只负责准入留痕和关键缺口提醒

快速会商只产出三种结果：

- 进入 focus_pool
- 进入 watchlist
- 拒绝准入

但对仓位层面，还应再多两种动作：

- `fill_cash_slot`
  - 当前有空余仓位，应作为补仓候选进入执行评估
- `replace_position`
  - 当前仓位已满或接近目标仓位，需要和持仓股直接比较，决定是否替换

## 十四、持仓纪律与替换规则

### 14.1 仓位纪律不是建议，而是团队目标

如果用户或系统治理层规定：

- 目标总仓 = `50%`

那么团队默认职责不是“随缘做到”。

而是：

1. 只要未达到 `50%`
2. 且风控没有给出明确阻断
3. 团队就有持续寻找机会并推动补足仓位的责任

这条规则会改变团队的工作心态：

- 不是“我只评论送到我面前的票”
- 而是“如果还有空仓，我有义务继续找值得占位的机会”

### 14.2 持仓不是放进去就不管

一旦有持仓，团队必须持续评估：

- 这只票是否仍值得占用仓位
- 它的逻辑是否被破坏
- 它是否只是“没犯错”，但已经不够优秀
- 当前是否出现了更值得替换进来的票

### 14.3 替换仓位的判断框架

新机会票要挑战现有持仓时，`ashare-strategy` 必须回答：

1. 新票强在哪里
2. 被替换票弱在哪里
3. 这是短期情绪更优，还是中枢逻辑更优
4. 替换后组合是否更集中、更有效

`ashare-risk` 必须回答：

1. 替换的交易成本是否值得
2. 被替换票是否已进入高风险区
3. 新票是否存在更差的流动性或回撤风险

`ashare-audit` 必须回答：

1. 这次替换是基于真实新证据，还是单纯追涨换票
2. 是否存在“频繁换仓导致纪律失真”的风险

### 14.4 允许持仓股的日内 T 机会

团队职责不应只覆盖“买不买”。

还应覆盖：

- 当前持仓股是否存在日内 T 的机会

这里的日内 T 不应交给单一 agent 决定。

建议分工：

- `ashare-runtime`
  - 提供分时、微观节奏、盘口与异动事实
- `ashare-strategy`
  - 判断这是趋势延续中的高抛低吸机会，还是噪音
- `ashare-risk`
  - 判断日内 T 是否会破坏整体风险纪律和交易成本纪律
- `ashare-audit`
  - 记录这类机会是否被频繁滥用

### 14.5 日内 T 的适用边界

不是所有持仓都适合做 T。

建议草案里明确：

- 龙头强趋势票：谨慎做 T，避免卖飞
- 高波动震荡票：允许更积极评估 T 机会
- 流动性差的票：默认不做 T
- 风险未解除的票：默认不做 T

## 十五、多策略全天候扫描

团队不能只围绕单一策略工作。

应把各类策略机会都交给对应角色持续观察。

至少包括：

- 竞价超预期
- 板块回流
- 龙头加速
- 分歧转一致
- 首板/一进二/二进三等情绪接力
- 低吸回封
- 盘中异动扩散
- 持仓股日内 T

### 15.1 策略工位建议

- `ashare-runtime`
  - 负责把系统和监控里出现的所有异常信号“看见”
- `ashare-research`
  - 负责判断事件驱动策略是否成立
- `ashare-strategy`
  - 负责把不同策略机会放进同一比较框架里排序
- `ashare-risk`
  - 负责判断不同策略机会对应的仓位和风险容忍度
- `ashare-audit`
  - 负责检查团队是否长期偏好某种策略而忽视其他机会

### 15.2 多策略不等于乱开仓

虽然各策略机会都可以被团队发现，但最终仍要经过：

- 统一候选比较
- 统一仓位纪律
- 统一风险闸门
- 统一留痕

所以这不是鼓励更多交易，而是鼓励更完整的机会覆盖。

## 十六、策略和权重不是固定神谕

### 14.1 团队必须有提策略与调权的权力

程序提供的是：

- 数据底座
- 基础策略
- 治理接口

团队不应该只是机械消费这些结果。

团队应该有权提出：

- 哪类机会今天更重要
- 哪类剧本今天应降权
- 哪类风险约束今天应收紧
- 哪类信号今天应优先

### 14.2 谁可以提治理建议

- `ashare-research`
  - 基于叙事、政策、题材结构变化提建议
- `ashare-strategy`
  - 基于排序命中率、强弱切换、剧本适配度提建议
- `ashare-risk`
  - 基于波动、回撤、执行失败和流动性恶化提建议
- `ashare-audit`
  - 基于盘后复盘和讨论质量提建议
- `ashare`
  - 把团队共识转成正式 proposal 或自然语言调参

### 14.3 治理建议分三层

- `intraday_temporary`
  - 只对当日会话生效
- `next_session`
  - 次日生效
- `structural`
  - 需要连续多日证据支撑

### 14.4 当前系统已有的承接点

现有系统已经有：

- `POST /system/adjustments/natural-language`
- `POST /system/params/proposals`
- `GET /system/agent-scores`
- `learning.score_state`
- `learning.prompt_patcher`
- `learning.registry_updater`
- `strategy.nightly_sandbox`

所以 OpenClaw 层不需要自己再发明一套治理系统。

OpenClaw 层真正要补的是：

- 谁能提
- 在什么时点提
- 提案是 preview 还是 apply
- 如何留痕
- 如何推送飞书

## 十七、盘后学习不是总结，而是教练组会议

### 15.1 每个角色都要形成学习产出

- `ashare-runtime`
  - 哪些信号出现得太晚
  - 哪些异动没有及时送入团队视野
- `ashare-research`
  - 哪些新闻是真催化，哪些是假热闹
- `ashare-strategy`
  - 哪些该抓的没抓，哪些不该推的推了
- `ashare-risk`
  - 哪些限制过严错失机会，哪些过松增加噪音
- `ashare-audit`
  - 哪些争议本该更早暴露，哪些定稿还不够扎实

### 15.2 盘后学习的正式输出

建议形成这些固定栏目：

- `missed_opportunities`
- `false_positives`
- `late_discoveries`
- `risk_overblocking`
- `unsupported_promotions`
- `proposal_candidates`

这些输出后续应该进入：

- `nightly_sandbox`
- `prompt_patcher`
- `registry_updater`
- `parameter proposals`

## 十八、这版草案新增的定稿问题

在原有 6 个讨论题基础上，再新增 4 个关键问题：

1. 团队是否正式进入“全天候值班”模式，而不是被动等主持人开会
2. 是否正式引入 `opportunity_ticket`，让新票准入有标准结构
3. 是否明确团队拥有策略/权重提案权，而不仅是用户调参权
4. 是否把盘后学习会的固定产出接到治理与自进化链路
5. 当目标仓位未满时，团队是否被赋予“持续补足目标仓位”的明确义务
6. 当出现更优机会时，团队是否被赋予“可替换现有仓位”的正式权力
7. 持仓股日内 T 是否纳入团队正式职责，而不是额外灵感行为
8. 多策略机会是否统一纳入全天候扫描，而不是只围绕单一候选逻辑

## 十九、当前更重要的结论

如果这些制度点不定，哪怕把会议 prompt 写得再漂亮，也还是一套“会说话的评审器”。

只有把：

- 持续值班
- 主动提票
- 仓位纪律
- 持仓替换
- 日内 T 管理
- 多策略覆盖
- 动态调权
- 盘后学习

这些制度立起来，这个项目才真正像一个全天候的智能量化团队。
