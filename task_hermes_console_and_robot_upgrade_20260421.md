## 一、现状诊断

### 1. 控制台展示层与业务主链错位

- `机会池` 当前主要消费 `/system/discussions/client-brief`，本质是讨论摘要，不是盘中机会流。
- `fast_opportunity_scan`、`runtime top picks`、`execution_precheck`、`execution_dispatch`、`supervision_board` 没有被统一聚合到主视图。
- 多个页面仍保留静态演示数据、占位文案或假按钮，不能真实反映当前交易态。
- 用户现在无法在一个页面里回答以下关键问题：
  - 今天为什么没买入
  - 某只候选为什么没进入执行池
  - 当前是卡在市场感知、讨论、风控还是执行桥
  - 哪个 Agent 超时、哪个环节需要催办
  - 当前持仓是否触发做 T / 止盈 / 止损 / 风险减仓建议

### 2. Hermes 仍是“关键词路由器”，不是“语义中控”

- 当前 `_resolve_command()` 仍以关键词匹配为主，只能把问题命中到 `workspace/sessions/profiles/schedules/ashare_overview` 等能力域。
- 程序没有强制把实时市场状态、最新 runtime、执行桥状态、监督阻塞、当前持仓上下文注入到回答链。
- 用户问“今天为什么没有买入操作”“那你启动啊”这一类自然语言时，容易落入帮助口径，而不是进入真实业务链。

### 3. 飞书入口职责混杂

- 当前一个飞书入口同时承担问答、催办、执行回执、参数调整、链接预览等职责，造成语义混杂。
- 三个 bot 在一个群内时，如果没有明确触发路由，会出现抢答、答非所问、消息洪泛。
- 当前真正缺的不是 QQ/微信通道，而是飞书内部的职责分层。

### 4. 外链依赖不可靠

- 当前外链地址不可通用访问，飞书卡片和机器人回复不能依赖点击外链才能理解核心情况。
- 在正式可达地址确定前，机器人必须优先“群内自解释回答”，链接只作为附加证据入口。

## 二、改造总目标

本轮改造完成后，系统必须具备以下能力：

1. 控制台首页直接展示全流程主链，而不是分散的功能页摘要。
2. 机会池不再只展示讨论摘要，而是真正聚合盘中机会流、runtime 候选、执行预检和淘汰原因。
3. Hermes 对自然语言问题具备“预设命令优先 + 基础 agent 理解兜底”的双通路，不再跳帮助页。
4. 飞书三机器人在同一群内职责清晰，问答、督办、回执分别收口。
5. 在没有统一公网入口前，机器人消息本身就能把结论、依据、卡点、下一步说明白。

## 三、目标架构

### 3.1 控制台目标结构

统一主视图按真实业务流程展开：

1. 市场态势
2. 盘中机会流
3. 候选入池与淘汰原因
4. Agent 讨论收敛与争议焦点
5. 风控预检与阻断原因
6. 执行派发与回执状态
7. 当前持仓盯盘与动作建议
8. 监督催办与履职状态
9. 全流程时间线

### 3.2 Hermes 目标结构

Hermes 改造成四层：

1. 意图识别层
2. 上下文注入层
3. 任务编排层
4. 结构化回答层

### 3.3 飞书三机器人目标结构

在同一群内先拆职责，不急于上 QQ / 微信：

1. `Hermes主控`
2. `Hermes督办`
3. `Hermes回执`

触发规则默认采用：

- `@Hermes主控`：问答、状态、机会、持仓、调参预览
- `@Hermes督办`：催办、ACK、作业追踪、超时升级
- `@Hermes回执`：预检、派发、成交、撤单、桥接异常
